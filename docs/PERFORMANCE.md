# 성능 계측 (Phase 0 측정 하네스 + 실측)

WASM 포팅 로드맵(`WASM-ENGINE-PORTING.md`)의 **Phase 0 = "눈 감고 포팅 금지,
측정 먼저"** 를 실제 도구와 숫자로 채운 문서. "틱이 어디로 가나"를 실측해서
다음에 무엇을 벡터화/포팅할지(그리고 **무엇을 안 할지**)를 데이터로 정한다.

## 1. 도구

두 개의 독립 하네스가 서로 다른 질문에 답한다.

### 1a. 인앱 패스별 프로파일러 (`?perf`)

- `src/game/engine/profiler.ts` — 프로세스 전역 싱글턴. 틱을 다섯 패스로 쪼개
  누적한다: **열(heat)·CA(물질 스캔)·오브젝트·드리프트·렌더.**
- `Simulation.step`과 `Game.frame`이 패스마다 `performance.now()` 구간을 재고,
  프레임 루프가 HUD 주기(0.5s)마다 롤링 평균을 `$perfPasses`로 발행한다.
- **개발 플래그**: URL `?perf`. 없으면 완전 무효(패스당 불리언 검사 하나, 계측
  호출 자체를 건너뜀 → 프로덕션 비용 0). 켜면 HUD 하단에 패스별 ms 줄이 뜬다.
- 열/CA/오브젝트/드리프트는 **틱당 ms**, 렌더는 **프레임당 ms**.

### 1b. 고정 벤치 장면 (`?bench=`)

`src/game/engine/benchScenarios.ts` — 재현 가능한 3종 장면(고정 PRNG, `Math.random`
미사용)을 시작 시 시드한다. 벤치 로드는 **휘발성**이라 저장 월드를 읽지도 쓰지도
않는다(오토세이브 차단).

| 장면 | 내용 | 무엇을 압박하나 |
|---|---|---|
| `empty` | 빈 그리드 | 상시 비용(빈 필드 열 스윕 + 드리프트 + 빈 화면 렌더) |
| `sparse` | 흩뿌린 모래 언덕 + 물웅덩이 + 작은 불 (~12% 점유) | **전형적 플레이** — 빈 공간 다수, 활성-타일 스캔의 주 대상 |
| `static` | 모래 언덕 + 물웅덩이(정착) | 점유 셀 대량, 전면 CA 스캔, 반응은 적음 |
| `active` | 나무+불 연소, 기름 화재, 용암-물 접촉 | 반응 다발 — CA 스캔/물질 update 규칙 스트레스 |

사용: `?perf&bench=active` 처럼 조합. `?fullscan`을 더하면 활성-타일 스캔을 끄고
전면 스캔으로 돌려 같은 기기에서 on-vs-off A/B를 잰다. (헤드리스로 `.perf` HUD
텍스트를 읽어 아래 표를 수집.)

### 1c. 커널 마이크로 벤치 (Node)

`wasm/bench/heat-bench.mjs` — 열확산 커널을 **JS 레퍼런스 vs Rust/WASM**으로
같은 입력에 대해 정상상태 처리량으로 맞붙인다(호스트처럼 버퍼 1회 할당·재사용,
틱당 전 substep). 실행: `node wasm/bench/heat-bench.mjs`.

## 2. 실측 (지표성 수치)

> 아래 절대 ms는 **CI 샌드박스 컨테이너 1회 측정치**라 기기마다 다르다.
> 가져갈 것은 **패스 간 비중**과 **JS↔WASM 비율**이지 절대값이 아니다.
> 하네스가 커밋돼 있으니 언제든 자기 기기에서 재측정하면 된다.

### 2a. 틱 패스별 비중 (`?perf&bench=…`)

| 장면 | 열 | CA | 오브젝트 | 드리프트 | 렌더/프레임 | 틱 합 |
|---|---|---|---|---|---|---|
| empty | 0.81 | 0.97 | 0.00 | 0.23 | 2.33 | 2.01 |
| static | 3.01 | **14.61** | 0.00 | 0.23 | 2.88 | 17.85 |
| active | 2.03 | **8.78** | 0.00 | 0.22 | 2.35 | 11.03 |

비중(틱 대비): 정적 = 열 17% / **CA 82%**, 활성 = 열 18% / **CA 80%**,
빈 = 열 40% / CA 48%.

### 2b. 열 커널 JS vs WASM (`heat-bench.mjs`)

| 그리드 | 셀 | JS ms/tick | WASM ms/tick | 속도향상 |
|---|---|---|---|---|
| 90×51 | 4,590 | 0.213 | 0.077 | 2.76× |
| 180×101 | 18,180 | 1.301 | 0.909 | 1.43× |
| 360×203 (기본) | 73,080 | 6.001 | 4.447 | **1.35×** |
| 600×400 | 240,000 | 19.880 | 13.830 | 1.44× |

작은 그리드가 유독 빠른 건 캐시에 다 들어가서다. 실사용 크기(73k~240k)에선
**스칼라 WASM이 JS 대비 ~1.35–1.44×**, 처리량이 ~50 Mcell·substep/s 부근에서
평탄 → 메모리 대역폭에 가까워지는 구간.

## 3. 해석 — 다음에 무엇을 (안) 할까

1. **CA 물질 스캔이 지배적이다(점유 장면 80%+).** 이는
   `WASM-ENGINE-PORTING.md` §3의 가설을 실측으로 확정한다: 최대 레버리지는
   WASM이 아니라 **활성 셀 / 더티 영역 추적**(빈·정적 영역 스캔 제외)이다.
   순수-JS 최적화이고 WASM 유무와 무관하게 유효.
2. **열은 2순위(점유 17–18%, 빈 40%)이고 이미 WASM(~1.35×).** 여기에 SIMD를
   더 얹으면 그 슬라이스가 대략 2배 → **틱 전체로는 한 자릿수 % 절감**에 그친다.
   게다가 SIMD f32는 **비트 동일이 깨져** 골든 테스트를 오차 허용으로 완화해야
   한다(§6 정책상 열은 허용). **가치는 있으나 CA 스캔보다 후순위.**
3. **렌더는 장면과 무관하게 ~2.3–2.9 ms/프레임 상수.** 다음 WASM 커널 후보로
   자주 거론되지만(픽셀 합성), 절대 비중이 CA 스캔보다 작다. 병목이 되기 전엔 급하지 않다.
4. **오브젝트·드리프트는 무시 가능**(이 장면들엔 자유 오브젝트 0, 드리프트
   ~0.22 ms 상수).

**결론(데이터 기반 우선순위)**: ① CA 스캔 활성-셀 추적(순수 JS, 최대 이득) →
② (원하면) 열 SIMD 승격(작지만 확실, 정확도 정책 완화 필요) → ③ 렌더 합성
커널은 병목화된 뒤. 즉 "SIMD부터"가 아니라 **"측정이 가리키는 CA 스캔부터"** 다.

## 4. 활성-타일 CA 스캔 (구현 + 실측)

§3 ①의 실구현. `updateCell`은 **빈·비겹침 셀에 대해 아무 일도 하지 않으므로**,
그런 셀만 담긴 타일은 건너뛰어도 결과가 완전히 동일하다. `engine/dirtyTiles.ts`가
그리드를 16×16 타일로 나눠, 활성 셀이 있는 타일만 스캔한다.

- **비트 동일**: 스킵되는 타일은 증명상 무해(inert)하고, 활성 셀은 전면 스캔과
  **완전히 같은 순서로** 방문된다. `test/active-tiles.ts`가 진짜 엔진(물질 89개·
  반응·겹침·열)을 전면 스캔 vs 타일 스캔으로 돌려 **매 틱 비트 동일**임을
  6장면·660틱·4중력에서 검증한다. 같은 시드 재실행으로 **결정론**도 함께 확인
  (멀티플레이 lockstep 전제).
- **관용적 불변식**: 모든 그리드 쓰기(`Grid.set`/`setOverlay`, SimContext의
  직접 버퍼 쓰기)가 자기 타일을 깨운다. 과다 표시는 성능만 약간 손해고, 누락만
  버그이므로 "쓰면 무조건 깨운다"가 안전한 기본값이다. 정착 더미는 스캔 중
  "활성 셀 있으면 재무장"으로 계속 깨어 있게 유지한다(점유 타일은 항상 스캔 —
  *점유하지만 정적인* 타일까지 재우는 건 별도 후속 과제, 아래).

### 실측 (on-vs-off A/B, `?fullscan`, 지표성)

| 장면 | CA 전면 | CA 타일 | 배속 |
|---|---|---|---|
| empty | 0.85 | 0.03 | **~30×** |
| sparse (전형) | 2.13 | 1.18 | **1.8×** |
| static (밀집) | ~19 | ~19 | ~1.0× (중립) |
| active (밀집) | ~8 | ~8 | ~1.0× (중립) |

즉 **빈 공간이 많을수록 크게 이득**(전형적 플레이가 여기), 꽉 찬 최악의 경우엔
중립(타일 순회 오버헤드 ≈ 스킵 이득). 밀집 장면 재무장 검사는 타일당 첫 활성
셀에서 단락(short-circuit)해 밀집에서도 셀당 비용을 안 문다.

### 남은 것 — Tier 2: *점유하지만 정적인* 타일 재우기

지금은 점유 타일을 항상 스캔한다(밀집=중립인 이유). 더 큰 이득은 정착해 아무
변화도 없는 점유 타일까지 재우는 것인데, 이는 **비트 동일이 아니다**: RNG로
자발 행동하는 물질(수명 감쇠, 불 깜빡임, 확률 반응)은 절대 재우면 안 되므로
물질별 "restless" 분류가 필요하고, 방해 전파에 1틱 지연이 생긴다. 재미>고증
프로젝트엔 대체로 무해하나 거동을 바꾸므로, **별도 PR + 승인** 대상으로 남긴다.
