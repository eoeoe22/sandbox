# WASM 기반 핵심 엔진 포팅 — 구상 단계 계획

> 상태: **구상(설계) 단계**. 이 문서는 "무엇을·왜·어떤 순서로" 옮길지 정하는
> 문서다. 아직 코드를 옮기지 않는다. 실제 착수 시 각 단계를 별도 PR로 낸다.
>
> 맥락: **1인 개발, 프로젝트 6일차.** WASM을 처음부터 하려다 한 번 미뤘고, 1차
> 기능이 마무리된 지금 "늦기 전에" 착수하려는 상황이다. 그래서 이 문서의 핵심
> 질문은 "지금 다 옮기자"가 아니라 **"시간이 지날수록 비싸지는 게 무엇이고,
> 지금 코드가 작을 때(≈17k줄) 무엇을 잠가둬야 나중에 손해를 안 보나"** 이다
> (→ §1.5). 협업 합의가 아니라 1인 판단이므로 편의성/재미의 트레이드오프는
> 본인이 바로 정하면 된다.

성능(1순위 재미를 지키기 위한 프레임 예산 확보)을 목표로, 시뮬레이션 핵심
엔진을 WASM/멀티스레드로 옮기는 로드맵을 정한다. 결론부터: **전면 재작성이
아니라, 측정 → 오프메인스레드 → 순수 수치 커널만 WASM → (조건부) CA 스캔**의
점진적 4단계로 간다. 아래에 그 근거와 대안, 위험을 정리한다.

**가장 먼저 안심할 점**: 손으로 쓴 물질 `update` 함수 89개는 **WASM 포팅
부채가 아니다.** 시뮬을 워커로 내려도 물질 함수는 **워커 안에서 그대로 JS로
실행**된다(워커에서 JS는 당연히 돈다). WASM은 열확산 같은 순수 수치 커널에만
쓴다. 즉 물질을 Rust/데이터로 다시 쓸 필요가 없고, 물질이 매일 늘어나는 것에
대한 "늦기 전에" 불안은 **해당 없음**이다 — 물질은 지금처럼 명령형 TS로
자유롭게 계속 추가하는 게 재미·개발 속도를 지키는 정답이다.

---

## 1. 현재 엔진 구조와 "이식 가능성" 지도

핵심은 이미 프레임워크 독립·평탄 TypedArray로 잘 분리돼 있다(`docs/ARCHITECTURE.md`).
포팅 관점에서 코드를 세 부류로 나누면:

| 구역 | 파일 | 성격 | 물질 콜백 | WASM 적합도 |
|---|---|---|---|---|
| **A. 순수 수치 커널** | `Simulation.diffuseHeat`, `driftBackground`, `Game.countParticles`, 렌더 픽셀 합성(`CanvasRenderer`) | 배열 in → 배열 out, 분기 단순, 데이터 병렬 | **없음** | ★★★ (SIMD 직행) |
| **B. CA 이동·이식 seam** | `SimContext`(tryMove/swap/move\*), `behaviors`(updatePowder/Liquid/Gas), `reactions` | 격자 인덱싱 + moved 가드 | 간접(경계) | ★★ (가능하나 B↔C 경계가 관건) |
| **C. 물질 규칙** | `materials/*.ts` **등록 물질 98개 중 89개가 custom `update`** (파일 106개 중 90개가 `update:`, 나머지는 인프라·디스패처) | 손으로 쓴 명령형 TS, ≈54개가 이웃 스캔, 대부분 `SimContext`로 콜백 | **자기 자신** | ★ (재작성=고비용·재미 리스크) |

핵심 관찰:

- **A는 물질 코드를 전혀 건드리지 않는다.** 특히 `diffuseHeat`는 틱당
  `HEAT_DIFFUSION_SUBSTEPS(=3)`회 전체 그리드를 4-이웃 부동소수로 도는, 이
  프로젝트에서 가장 크고 규칙적인 단일 루프다. **가장 안전하고 이득이 큰 첫 WASM 대상.**
- **C가 "엔진의 재미"의 실체다.** 물질 추가 = 파일 하나(2순위 편의성)라는
  1인 개발 속도가 여기 걸려 있고, 89개 함수는 `blast` 연쇄·증류·연소 전선처럼
  다단계·비정형이라 데이터로 환원되지 않는 것이 많다. **여기를 Rust나 데이터
  VM으로 전면 재작성하면 편의성과(미묘한 거동 차이로) 재미가 함께 위험해진다.**
  다행히 §1.5의 결론대로 **C는 WASM으로 옮기지 않아도 된다**(워커 안 JS로 유지).
- RNG는 전부 `SimContext.chance/randInt`를 통과한다(물질 내 직접 `Math.random`
  0회). 좋은 seam이지만, 스레드/WASM으로 넘어가면 **RNG 결정론·시드 관리**가
  새 과제가 된다(§6).
- `objects.ts`(2064줄)의 강체 물리는 그리드 옆에서 도는 별도 패스라 1차 범위에서 제외한다.

### 냉정한 전제: WASM ≠ 자동으로 빠름

잘 튜닝된 **단형(monomorphic) JS + TypedArray**는 이미 매우 빠르다. WASM이
실제로 이기는 지점은 사실상 둘뿐이다: **(1) SIMD**(수치 커널 벡터화),
**(2) 스레드**(메인 스레드에서 내리기). 그 외의 "그냥 WASM으로 옮기면 빠르겠지"는
경계 넘나듦(JS↔WASM call overhead)·직렬화 비용으로 오히려 손해 볼 수 있다.
그래서 **측정 먼저, 그리고 A(수치 커널)부터**가 원칙이다.

## 1.5. "늦기 전에" — 무엇이 시간이 지날수록 비싸지나 (1인 개발 핵심)

WASM 착수의 동기가 "늦기 전에"라면, **정말로 복리로 비싸지는 항목**과 **언제 해도
난이도가 같은 항목**을 구분하는 게 먼저다. 세 가지로 나뉜다:

| 항목 | 시간이 지나면 | "늦기 전에" 해당? |
|---|---|---|
| **워커/버퍼 경계**(오프메인스레드) | 메인 스레드에서 그리드를 **동기적으로** 만지는 코드(`PointerPainter`, `brushTools`, inspect, `persistence`)가 늘수록 분리가 비싸짐 → **복리 증가** | ✅ **이게 진짜 시간제한** |
| **수치 커널 WASM**(열확산 등) | 자족적 루프. 물질이 98개든 500개든 난이도 동일 | ❌ 언제 해도 같음(evergreen) |
| **물질 → 데이터 VM**(3b) | 명령형 물질이 쌓일수록 변환 대상↑ | ⚠️ 유일하게 해당하나 **재미/속도상 비권장** |

핵심 결론 세 가지:

1. **정말 "늦으면 손해"인 건 워커 경계 하나뿐이다.** 지금은 6일차라 그리드를
   동기적으로 만지는 지점이 몇 곳뿐(`PointerPainter`가 직접 `grid.set/setTemp`).
   기능이 늘수록 이런 지점이 늘어 워커 분리가 점점 비싸진다. **저비용으로 잠가둘
   가치가 가장 큰 게 이 경계.** 지금 당장 완전 분리를 안 하더라도, 최소한 **새
   코드가 그리드를 직접 만지지 말고 seam(명령/메시지)을 통하게 하는 규율**만
   들여도 미래 분리 비용이 급감한다.
2. **수치 커널 WASM은 서두를 이유가 없다** — 다만 "코어 엔진을 WASM화"라는 원래
   하고 싶던 그림을 가장 깔끔하게 만족시키는 첫 결과물이라, 만족감·저위험 측면에서
   좋은 첫 착수점이긴 하다. 급해서가 아니라 하고 싶어서 먼저 해도 된다.
3. **물질 89개는 영영 WASM으로 안 옮겨도 된다** → "늦기 전에" 패닉의 대상이
   아니다. 데이터 VM(3b)은 이 패닉을 유일하게 정당화하는 길이지만, 그 대가로 재미의
   원천(명령형 물질 파일 하나로 뚝딱 만드는 자유)을 깎으므로 1인 재미 프로젝트엔
   맞지 않는다. **VM을 만들지 않기로 지금 결정하면, 물질을 계속 자유롭게 늘려도
   포팅 부채가 0으로 유지된다.**

즉 로드맵의 시간 축은 이렇게 읽는다: **워커 경계는 (지금 규율로) 일찍,
수치 커널 WASM은 (하고 싶을 때) 아무 때나, 물질 VM은 (아마) 영영 안 함.**

## 2. 단계별 로드맵

### Phase 0 — 측정 하네스 (선행 필수, WASM 아님)

무엇을 옮길지 정하기 전에 **어디서 시간을 쓰는지** 봐야 한다. 눈 감고 포팅 금지.

- 틱을 구간별로 계측: `diffuseHeat`(×substeps) / CA 스캔(updateCell 루프) /
  `stepObjects` / `driftBackground` / 렌더. `performance.now()` 누적 + HUD의
  기존 frameMs 옆에 패스별 ms를 (개발 플래그로) 노출.
- 시나리오 3종 고정 벤치: ① 빈 그리드 ② 물·모래 반쯤 채운 정적 더미
  ③ 불·폭발 등 활성 반응 다발. 셀 예산도 상·하한(360×203 ~ 600k)에서.
- **산출물**: "열확산 vs CA 스캔 vs 렌더" 실측 비중표. 이 표가 Phase 2/3의
  우선순위와 "그만둘 근거"를 준다.

> 예상(가설, Phase 0가 검증): 정적 장면은 열확산·렌더가, 활성 장면은 CA 스캔이
> 지배적. 열확산은 substep 3회라 정적에서도 무시 못 할 것으로 본다.

### Phase 1 — 오프메인스레드 (Web Worker, WASM 아님)

시뮬레이션을 통째로 **Web Worker**로 내린다. 틱당 속도를 올리진 않지만,
렌더/입력(UI) 스레드를 시뮬에서 분리해 **입력 반응성·렌더 프레임을 지킨다.**
그리고 이것이 이후 어떤 WASM 코어를 붙이든 **공통 전제**다(WASM 코어는 워커
안에서 도는 게 자연스럽다).

- **버퍼 공유 방식**: `Grid`의 평탄 TypedArray들을 `SharedArrayBuffer` 위에
  올려 워커=시뮬, 메인=렌더가 **제로카피**로 본다. (SAB는 COOP/COEP
  crossOriginIsolated 헤더가 필요 — Cloudflare Workers Static Assets에서
  `_headers`로 설정 가능한지 §5에서 확인.)
  - SAB가 막히면 대안: **더블 버퍼 + `postMessage(transfer)`** 로 그리드
    스냅샷을 프레임마다 넘긴다(약간의 복사비용, 헤더 불필요).
- 동기화: 시뮬은 고정 틱, 렌더는 최신 스냅샷을 읽음(찢김은 저해상 픽셀아트라
  실무상 무해, 필요 시 seqlock/더블버퍼).
- 입력(브러시)·설정(중력/경계/스모크)은 워커로 메시지 전달. `PointerPainter`가
  직접 그리드를 쓰는 현재 구조(`grid.set/setTemp/setAux/setTint`)는 "브러시 명령을
  워커로 보내기"로 바꾼다 — **입력 배관(plumbing)은 이 단계에서 손대는 유일한
  UI측 변경**이다.
- **물질 규칙·렌더 코드는 변경 없음.** `Grid`/`Simulation` 소유권만 워커로
  이동하고, 물질 `update` 함수 89개는 워커 안에서 JS 그대로 실행된다. 바뀌는 건
  코어 소유권과 입력 배관뿐, 물질/렌더 로직은 불변.
- **위험**: 저지연 브러시 페인팅이 메시지 왕복으로 한 틱 늦어질 수 있음 →
  낙관적 로컬 프리뷰 또는 SAB 직접 쓰기로 완화.

### Phase 2 — 순수 수치 커널 WASM화 (구역 A)

Phase 0가 가리키는 A 커널을 **하나씩** WASM으로 교체한다. 인터페이스가
`(배열 포인터, w, h, params) → 배열` 로 깔끔하고 물질 콜백이 없어 위험이 낮다.

- **1순위: `diffuseHeat`.** `temp`·`tempScratch`·`cells`(→conductivity LUT)만
  넘기면 되는 자족 루프. WASM SIMD로 4-이웃 확산을 벡터화. **JS 원본은 레퍼런스로
  남겨** 동일 입력에 대해 출력을 부동소수 오차 범위로 diff 검증(골든 테스트).
- 후속 후보: 렌더 픽셀 합성(cells+temp+tint+overlay → Uint32 프레임버퍼),
  `countParticles`, `driftBackground`. 각각 독립적으로 켜고 끌 수 있게.
- **툴체인 선택**(§4): 1차는 **AssemblyScript**(TS 생태계 유지, 진입장벽 낮음)로
  빠르게 검증, SIMD 이득이 확실하면 커널만 **Rust+wasm-bindgen**으로 승격 고려.
- **롤백 안전장치**: 각 커널은 `USE_WASM_HEAT` 같은 플래그로 JS 폴백. WASM
  미지원/로드 실패 시 자동 JS. 빌드는 Vite `?init`로 정적 산출물에 포함.
- **기대 이득**: 열확산이 실측 비중 상위면 substep 3회가 SIMD로 몇 배 빨라짐 →
  같은 예산에서 셀 예산 상향 또는 substep 상향(열 리얼리즘) 여지.

### Phase 3 — CA 물질 스캔 (구역 B/C, **조건부·연구 과제**)

Phase 0가 "활성 장면에서 CA 스캔이 지배적이고 A 최적화로 부족"임을 보였을
때만 착수. 세 가지 길이 있고 각각 트레이드오프가 크다:

- **3a. 이동 primitive만 WASM + 물질은 JS 유지**
  스캔 루프가 B(공통 이동)와 C(물질별 update)를 매 셀 교대 호출하므로, WASM
  코어가 custom 물질을 만날 때마다 **JS로 콜백**해야 한다. 셀 단위 경계
  넘나듦이 이득을 대부분 상쇄 → **비추천**(98개 중 89개가 custom이라 콜백이 지배적).
- **3b. 데이터 기반 물질 VM (권장 연구 방향)**
  `reactions` 테이블(이미 3개 사용)과 phase 기본 거동을 확장한 **작은 규칙
  집합/opcode**로 물질을 데이터로 표현하고, 그 VM을 WASM에서 실행. 성공하면
  "물질 = 데이터 한 줄" 로 **편의성이 오히려 강해지고** WASM 안에서 돈다.
  - 한계: 다단계·비정형 로직(`blast` 질량 서베이, 석유 증류 사슬, 연소 전선,
    `objects` 연동)은 데이터로 안 떨어짐 → **하이브리드**(단순 물질은 VM/WASM,
    복잡 물질은 JS 유지)로 가야 하고, 그럼 3a의 경계 문제가 부분적으로 재등장.
  - 접근: 먼저 순수 JS에서 "물질을 최대한 데이터로 환원"하는 리팩터를 해보고
    (WASM 없이도 단형 dispatch로 빨라짐), 환원 가능 비율을 측정한 뒤 WASM VM
    가치를 판단.
- **3c. 물질 로직 전면 Rust 재작성**
  최대 성능, 최대 비용. **2순위 편의성(기여 모델)과 1순위 재미(거동 동일성)를
  정면으로 위협** → 이 프로젝트 가치관상 **비권장.** 특정 초-핫 물질 소수만
  선별 재작성하는 정도로 제한.

> **Phase 3 기본 입장**: 착수 전 반드시 "값싼 순수-JS 승리"(§3)를 먼저 소진.
> CA 스캔의 최대 실질 이득은 WASM이 아니라 **활성 셀/더티 영역 추적**(빈 공간을
> 아예 스캔 안 함)일 가능성이 높다.

## 3. 먼저 시도할 값싼 순수-JS 최적화 (WASM보다 우선)

WASM은 비용이 크다. 아래는 저비용·저위험이며 WASM 이득과도 곱해진다:

- **활성 셀 / 더티 영역 추적**: 정적·빈 영역을 스캔에서 제외(active bitset 또는
  타일 단위 dirty flag). 대부분의 장면은 대부분이 정적이라 **단일 최대 이득**
  후보. WASM 유무와 무관하게 유효.
- **dispatch 단형화**: `getMaterial(id).update?.()` 간접 호출을 phase별로 묶어
  megamorphic 콜사이트를 줄임(예: phase 기본 물질은 스위치로 인라인).
- **열확산 조기 종료**: 평형에 도달한(이웃과 온도차 무시할) 타일은 substep 건너뜀.
- **RNG 비용 절감**: `Math.random` 호출 수 축소(배치 생성, xorshift).

이 목록을 Phase 0 벤치로 각각 검증 → 이득/노력 비로 정렬.

## 4. 툴체인 후보

| 후보 | 장점 | 단점 | 권장 용도 |
|---|---|---|---|
| **AssemblyScript** | TS와 문법 유사, 팀 진입장벽↓, TypedArray 친화 | 최적화가 Rust보다 얕을 수 있음 | Phase 2 **1차 검증**, 데이터 VM 프로토타입 |
| **Rust + wasm-bindgen** | 최고 성능, 성숙한 SIMD, wasm-opt | 툴체인·빌드 복잡, 언어 장벽 | 이득 확인된 **핫 커널 승격**, 3c 선별 |
| **C/Emscripten** | 성숙 | 빌드 무거움, 정적 배포와 궁합 애매 | 비권장 |

**전략**: AssemblyScript로 싸게 프로토타이핑 → 골든 diff로 정확성 확보 →
SIMD 이득이 실측되는 커널만 Rust로 승격. 툴체인을 하나로 못 박지 말고
**커널 단위로** 결정.

## 5. 배포·인프라 확인 사항

- **crossOriginIsolated(SAB 전제)**: `SharedArrayBuffer`는 COOP
  `same-origin` + COEP `require-corp` 헤더가 필요. Cloudflare Workers Static
  Assets에서 응답 헤더 주입 가능한지(`_headers` 또는 Worker 스크립트) **Phase 1
  착수 전 확인**. 불가 시 transfer 더블버퍼 경로로 우회(§Phase 1).
- **WASM 번들**: Vite `?init`로 `.wasm`을 정적 산출물에 포함(현재 `astro.config`
  주석이 이미 이 경로를 예고). Cloudflare 정적 자산은 `.wasm` MIME 서빙 확인.
- **PR 프리뷰 파이프라인**: 기존 `wrangler versions upload` 프리뷰가 WASM 자산
  포함 빌드에서도 그대로 도는지 첫 Phase 2 PR에서 검증.
- **폴백**: WASM/스레드 미지원·헤더 미적용 환경에서 **항상 순수-JS 경로로 자동
  강등**(기능 저하 없이 느려질 뿐). 이게 "정적 파일만 배포"의 단순함을 지킨다.

## 6. 정확성·결정론 리스크

- **거동 동일성(1순위 재미 보호)**: 각 WASM 커널은 JS 레퍼런스와 **골든 테스트**로
  묶는다. 열확산은 부동소수라 비트 동일이 아닌 **오차 허용 diff**, 정수 로직은
  비트 동일 기대.
- **RNG**: 현재 `Math.random` 기반. WASM/워커로 가면 RNG를 **명시 시드 xorshift/
  PCG로 교체**하고 seam(`SimContext.chance/randInt`)만 바꾸면 물질 코드는 불변.
  이는 재현 가능한 버그 리포트·리플레이라는 부수 이득도 있음.
- **저장 포맷 호환**: `persistence.ts`가 `cells/temp/aux/overlay/overlayAux`를
  저장. 버퍼 레이아웃을 SAB로 옮겨도 **직렬화 포맷은 불변**으로 유지(로드 시 복사).
- **스캔 순서 의존성**: moved 가드가 순서 독립성을 보장하지만, 워커 병렬화(타일
  분할) 시 경계 셀 경합이 생김. 1차는 **단일 워커(직렬 스캔)** 유지, 타일 병렬은
  훨씬 뒤 별도 연구.

## 7. 권장 실행 순서 (요약)

시간 축(§1.5)을 반영한 순서. "늦기 전에"인 워커 경계는 규율로 일찍, 수치 커널
WASM은 하고 싶을 때, 물질 VM은 아마 영영 안 함.

0. **지금 당장(코드 규율)** — 새 코드가 그리드를 직접 만지지 말고 seam을 통하게
   유지. 워커 분리를 나중에 해도 비용이 안 커지도록 표면적을 묶어둔다. (문서화·습관,
   별도 PR 불필요)
1. **Phase 0 측정** — 패스별 ms 하네스 + 3종 벤치. (WASM 아님, 필수 선행)
2. **§3 값싼 JS 승리** — 활성 셀/더티 영역 추적부터. 이득/노력 순 적용.
3. **Phase 1 오프메인스레드** — 단일 Web Worker + (SAB 또는 transfer 더블버퍼).
   §1.5 기준 **유일하게 시간에 쫓기는 항목**이라, 코드가 더 커지기 전에 하는 게 이득.
4. **Phase 2 수치 커널 WASM** — `diffuseHeat` 먼저, 골든 diff + JS 폴백 플래그.
   원래 하고 싶던 "코어 WASM화"의 첫 결과물. 서두를 필요는 없고 하고 싶을 때.
5. **Phase 3 (조건부)** — Phase 0/§3 이후에도 CA 스캔이 지배적일 때만.
   3b 데이터-VM은 재미·속도 대가가 커 **아마 영영 안 함**, 3c 전면 재작성은 비권장.

각 단계는 **독립 PR**, 각 PR은 **JS 폴백 유지**로 언제든 되돌릴 수 있게 한다.
Phase 1과 2는 순서를 바꿔도 되지만(둘 다 물질 코드 불변), 1을 먼저 하면 2의 WASM
커널이 자연스럽게 워커 안에 자리 잡는다.

## 8. 열린 결정 사항 (1인 판단)

협업 합의가 아니라 본인이 바로 정하면 되는 것들. 현재 기울어진 방향을 함께 적는다:

- **목표 지점**: "Phase 2까지(수치 커널 WASM)"로 충분한가, "Phase 3 데이터-VM"까지
  갈 것인가. → 재미·개발속도를 1·2순위로 두는 이 프로젝트에선 **Phase 2까지 권장,
  VM은 안 함** 쪽.
- **SAB vs transfer**: Cloudflare에서 COOP/COEP 헤더를 켜 `SharedArrayBuffer`
  제로카피로 갈지, 헤더 없이 transfer 더블버퍼로 갈지. → 헤더 적용이 `_headers`로
  간단하면 SAB, 막히면 더블버퍼(저해상 픽셀아트라 복사비용 무해).
- **툴체인 1차**: **AssemblyScript로 시작(권장)** — TS와 문법이 가까워 1인
  진입장벽이 낮다. SIMD 이득이 실측된 커널만 나중에 Rust로 승격.
