# 검토 제안 (Proposals)

> 엔진 확인 결과(types/behaviors/SimContext/Grid/Simulation/registry 실측): `register({...})` 한 번으로 물질이 등록되고, `update`를 생략하면 `phase`의 기본 거동이 자동 부착됩니다(Solid/Empty는 update 없음 = 정적). 이동은 `tryMove`의 밀도 기반 치환(무거운 것이 가벼운 유체/가스를 밀어냄)으로 처리되고, 스캔은 바닥→위·좌우 교대이며 `moved` 가드가 한 틱 이중 처리를 막습니다. 밀도 기준선: 물 3 · 모래 5. 물질 추가 비용은 실제로 "파일 하나 + index.ts 2줄"입니다.
>
> **중요한 공통 함정:** `SimContext.set()`은 `swap()`과 달리 `moved`를 세우지 않습니다. 그래서 커스텀 update가 위쪽(아직 스캔 안 한) 셀을 `set`으로 만들면 같은 틱에 재처리되어 폭주(불이 순식간에 번짐, 덩굴이 한 프레임에 다 참, 폭발 연쇄 타이밍 붕괴)합니다. 불·폭약·식물 계열의 전제 조건은 아래 [공통 인프라](#a-3-공통-인프라-먼저-깔면-좋은-것)입니다.

## A. 새 물질 아이디어

제안자별 중복(불×4, 수증기×4, 화약/폭약×4, 소금×3, 돌×2, 산×2)은 병합했습니다. `engineFit` 기준 두 그룹으로 나눕니다.

### A-1. 지금 바로 구현 가능 (pure-data / custom-update, 새 필드 불필요)

| 물질 | phase | 거동 / 상호작용 | engineFit | 난이도 |
|---|---|---|---|---|
| **돌 / 바위** (Stone) | Solid | 정적 장벽(update 없음, 벽의 색·의미 변형). 물·모래가 위에 고임·퇴적. "용암+물→돌" 반응의 산물이자 산 부식의 대상. | pure-data | easy |
| **연기** (Smoke) | Gas | 위로 확산 후 확률 소멸. 불의 소멸 산물로 쓰는 동반 물질. | pure-data | easy |
| **소금물** (Saltwater) | Liquid | 담수보다 밀도 약간 높게(≈4) → 물 아래로 가라앉음. 소금 용해의 결과물. | pure-data | easy |
| **소금** (Salt) | Powder | 낙하·퇴적(밀도 4: 물 3<소금<모래 5, 물속에 가라앉음). 물 인접 시 확률적으로 자신→빈칸, 물→소금물로 용해. 염도는 추적 안 함(단순화). | pure-data(낙하) → custom-update(용해) | medium |
| **수증기** (Steam) | Gas | 상승·확산(밀도 최하). 매 틱 낮은 확률로, 특히 천장·벽에 막혀 정체하면 확률↑ 응결→물. 물+열원(불/용암)에서 생성 → 물↔수증기 순환의 한 축. | custom-update | medium |
| **불** (Fire) | Gas | 상승·대각상승·깜빡임(밀도 매우 낮음). 이웃 8칸 가연물(기름·화약·식물·불씨)을 확률 점화, 자신은 매 틱 확률 소멸→빈칸(또는 연기). 물 인접 시 즉시 소화, 닿은 물은 수증기로. 벽·모래는 불연물. | custom-update | medium |
| **용암** (Lava) | Liquid | 확률 게이트로 느리게 흐름(밀도 4). 물 인접 시 자신→돌, 물→수증기. 가연물 점화(발화원). 물은 위로 뜨고 모래는 가라앉음. | custom-update | medium |
| **산** (Acid) | Liquid | 흐름(밀도>물, 바닥부터 부식). 이웃 부식물(돌·모래·소금 등)을 확률적으로 빈칸으로 제거하고 자신도 확률 소모(무한 부식 방지). 벽·유리는 내산성. | custom-update | medium |
| **화약 / 폭약** (Gunpowder) | Powder | 평소 낙하·퇴적. 불·용암·점화된 화약 인접 시 기폭 → 반경 R 셀을 불/빈칸으로 `set`, 자신 소멸, 인접 화약 연쇄. 물에 젖으면 불발. 벽/돌은 반경을 막아 폭발 성형. | custom-update | hard |
| **니트로글리세린** (Nitro) | Liquid | 흐르고 고임(물보다 무거움). 불·불씨·폭발 접촉 시 화약보다 큰 대반경 즉시 기폭·연쇄. *원작의 "낙하·충격만으로 폭발"은 속도장(velocity) 필요 → 여기선 접촉·열 기폭으로 대체.* | custom-update | hard |
| **씨앗·덩굴** (Seed/Plant) | Powder(씨앗)+Solid(덩굴) | 씨앗은 낙하·안착, 물 있으면 발아. 덩굴은 인접 물을 소비하며 위쪽 빈칸으로 확률 성장, 물 고갈 시 멈춤. 불의 연료. *성장 상한은 물 소비로 자연 제한(life 없이도 가능).* | custom-update | hard |

**핵심 5종 — 현재 엔진 구현 한 줄:**
- **불**: `updateGas` 상속으로 상승 + 이웃 가연물 확률 `set`(점화) + 자기 확률 소멸. 수명 필드 없이 무기억 확률 소멸(깜빡임은 감수).
- **소금**: `updatePowder` 상속 + 4방향 물 인접 시 확률적으로 자신 빈칸·물 셀 소금물로 교체.
- **수증기**: `updateGas` 상속 + 매 틱(또는 정체 시) 확률 응결로 자신을 물로 교체.
- **돌**: Solid로 `register`만 호출 — update 없이 완성(가장 싼 물질). 자갈이 필요하면 초고밀도 Powder 변형으로도 순수 데이터.
- **폭약**: `updatePowder` 상속 + 이웃 불 감지 시 중첩 루프로 반경 셀을 불로 `set`하고 자신 소멸(연쇄).

### A-2. 엔진 확장이 있어야 제맛 (온도장 temp / 수명 life / 속도 velocity)

| 물질 | phase | 거동 / 상호작용 | engineFit | 난이도 |
|---|---|---|---|---|
| **불씨** (Ember) | Powder | 낙하·퇴적하며 바닥 화약 더미로 옮겨 붙는 "도화" 역할. 핵심인 "서서히 식어 재로"는 `life`(냉각 카운트다운) 또는 `temp`(발광색 그라데이션)가 정석. *확률 냉각으로 임시 구현은 가능.* | needs-new-mechanic | medium |
| **전기(스파크)·도선** (Spark/Wire) | Solid(도선)+신호 상태 | 도선 따라 신호 전파 → 화약 원격 기폭·회로 놀이. id 하나로는 도선 전체가 즉시 켜져 회로가 성립 안 됨. **불응(refractory)** 처리를 위한 `life`(또는 상태) 필드 + 더블버퍼(동시 갱신) 필요. 예약된 `life` 확장점의 대표 사용처. | needs-new-mechanic | hard |

> **부분 확장이 도움 되는 경우(선택):** 수증기의 "실제 끓는점/이슬점" 판정과 불의 "고정 연소시간·열 전파"는 `temp`/`life`가 있으면 훨씬 자연스럽지만, 위 A-1의 확률 근사 버전으로도 충분히 플레이 가능합니다. 니트로의 "충격 기폭"만이 예약에 없는 `velocity` 필드를 새로 요구합니다.

### A-3. 공통 인프라 (먼저 깔면 좋은 것)

여러 물질이 같은 두 가지를 요구합니다. 물질 추가 규칙("파일1+2줄")을 깨지 않는 순수 데이터/헬퍼 확장입니다.

- **`spawn(x, y, id)` 헬퍼** — `set` + 대상 셀 `moved=1`을 함께 세우는 생성 함수. 불·화약·니트로·덩굴이 위쪽/주변 셀을 만들 때 같은 틱 폭주를 막는 안정 장치. **불·폭약·식물 계열의 사실상 전제 조건.**
- **데이터 태그 `flammable?: boolean`** (로직 아닌 데이터 확장) — 불이 가연성을 하드코딩된 id 목록이 아니라 태그로 판별하게 해 확장성↑. 마찬가지로 산의 "내산성", 폭발의 "차폐" 같은 태그도 데이터로.
- **예약 필드(`temp`/`life`/`velocity`)는 별도 마일스톤** — Grid 22–25행 주석에 예약돼 있고, A-2 물질과 정교한 상전이를 원할 때만 도입.

## B. 물질 팔레트 UI 구상

세 접근법 모두 **공통 토대**로 수렴합니다: `Material`에 optional `category?: string` 한 줄만 추가(생략 시 `phase`에서 자동 유도 → 기본 분류는 스키마 변경 0), 지우개(id 0)는 탭에서 빼 상단 고정 도구로, `$activeCategory` atom 하나 추가, 데스크톱↔모바일은 미디어쿼리로 전환. "파일1+2줄" 규칙은 세 안 모두 유지됩니다.

### 세 접근법 비교

| | ① 사이드바 + 접이식 아코디언 | ② 하단 작업표시줄(Taskbar) | ③ 카테고리 탭 |
|---|---|---|---|
| **형태** | 좌상단 패널 유지, `<details>` 그룹 + 팔레트만 내부 스크롤 | 물질 선택을 화면 하단 전폭 바로 분리(탭 행 + 가로 아이콘 스트립) | 팔레트 상단 탭(고체·가루·액체·기체·특수), 활성 탭 물질만 그리드 |
| **핵심 장점** | 네이티브 `<details>`로 JS/ARIA/키보드 거의 공짜, 기존 UX·근육 기억 계승, 파괴적 변경 없음 | 엄지 도달성 최고(모바일·한 손), 캔버스 해방, 그리는 중 빠른 전환, 게임 관습(핫바) | phase가 이미 존재 → 기본 4탭 **스키마·엔진 변경 0**, 라벨=거동이라 멘탈모델 일치, 국소 수정 |
| **핵심 단점** | 접힌 그룹은 클릭 한 번 더, 다중 열림 시 세로 길어짐, `<details>` 애니메이션 커스터마이즈 번거로움 | 하단 바가 입자 쌓이는 바닥을 가림(공간 예약 필수), 2탭 내비, 넓은 화면에선 좌우 여백 낭비 | 한 phase에 물질이 쏠리면(액체 20개) 탭 안에서 다시 스크롤, "특수"는 결국 category 필드 필요 |
| **비용** | 낮음(+`categories.ts`) | 중간(새 `MaterialTaskbar`, 셸 마운트, 제어 HUD 분리) | **가장 낮음**(MaterialPalette 한 파일 + atom 1개) |

세 안 모두 **20~40개 확장을 견디고, 40개를 넘거나 한 축에 쏠리면 상단 검색/필터를 병행**해야 한다는 데 동의합니다. 지우개 고정, 즐겨찾기/최근 사용 핀도 공통 완화책입니다.

### 최종 권장: 하이브리드 (③을 뼈대로, 반응형으로 ②를 겹침)

동일 마크업을 미디어쿼리로 재배치하는 것이 가장 저렴하고 두 세계를 모두 얻습니다.

- **데스크톱(넓은 폭) = 좌상단 사이드바 안의 카테고리 탭(③).** phase 자동 유도로 기본 4탭이 공짜, 폭은 200~220px로 살짝 넓혀 2~3열 스와치 그리드(`auto-fill, minmax(44px,1fr)`). 탭 전환은 `$activeCategory`만 바꾸고 실제 선택은 유지, 그리드 하단에 "현재: Sand" 캡션 필수. 재생/스텝/브러시/FPS 등 시뮬 제어는 사이드바에 그대로.
- **모바일(세로 · `max-width:640px` 또는 portrait) = 하단 도크(②).** 팔레트만 별도 래퍼로 빼 `matchMedia`로 하단으로 이동: (1) 탭 칩 가로 스크롤, (2) 지우개 고정 + 활성 탭 스와치 가로 스크롤(터치 타깃 44~48px), (3) 축약 제어 바(▶/⏭/브러시/⚙). 캔버스 컨테이너에 `padding-bottom`으로 바 높이 예약(오버레이 아님)해 바닥 셀이 안 가리게, `env(safe-area-inset-bottom)` 여백.
- **공통 완화책:** 팔레트 상단 **검색 input 1개**(40개 근처에서 특히 유효, 있으면 탭 무시하고 평면 필터). "특수" 탭(불/산/화약)만 `category: '특수'` 한 줄로 phase 교차 분류. 빈 그룹 탭은 자동 숨김 → 물질 4개~40개까지 같은 컴포넌트가 매끄럽게 확장.

아코디언(①)은 좋은 대안이지만, phase 탭이 이 저장소에선 "가장 싸게" 스케일링되고(스키마 0) 탭당 5~12개만 노출해 세로 폭주를 막으므로 뼈대로는 ③이 우세합니다. 물질이 계속 10개 안팎이면 세 안 모두 과설계이니 현재 평면 리스트 유지가 낫습니다.

## 다음 단계

우선순위 (위험 낮음·의존도 순):

1. **공통 인프라 먼저.** `spawn(x,y,id)`(set + moved) 헬퍼와 optional `category?` / `flammable?` 데이터 태그 추가 — 이후 거의 모든 반응성 물질의 전제.
2. **저위험 물질로 팔레트 확장 검증.** 돌(pure-data, easy) + 연기/소금물(pure-data 동반)을 넣어 물질 추가 파이프라인과 UI 스케일을 먼저 확인.
3. **고전 상호작용 루프 완성.** 산(독립적, medium) → 물↔수증기 순환(수증기+불) → 용암(불–물–돌–수증기 조합). 여기서 반응 패턴이 안정화됨.
4. **폭발 계열(hard).** 화약/폭약을 spawn 헬퍼 위에서 구현, 그 다음 니트로. 반경 `set` + 연쇄 타이밍을 moved 가드로 검증.
5. **UI는 ③부터.** 카테고리 탭(스키마 0) → 검색창 → 모바일 하단 도크 미디어쿼리 순으로 점진 도입.
6. **엔진 필드 확장은 마지막 마일스톤.** 불씨·전기(스파크)·정교한 온도 기반 상전이를 원할 때 `life`/`temp`(필요 시 `velocity`)를 별도로 추가.
