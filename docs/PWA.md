# PWA · 오프라인 지원

브라우저에서 "홈 화면에 추가/설치" 가능한 PWA로 만들고, 설치 후에는 오프라인에서도
앱이 열리도록 한다. 핵심 제약: **`astro.config.mjs`의 콘텐츠 해시 파일명 정책
(`docs/ARCHITECTURE.md` "정적 파일 해시 & 빌드 난독화" 참고)과 절대 충돌하면 안
된다** — 서비스 워커가 오래된 해시 자산을 붙들고 있어서 새 배포본이 안 보이는
사고가 나면 안 된다는 뜻.

## 구성 파일

모두 `public/`에 있다 — Vite가 처리하지 않고 그대로 복사되어 **해시가 붙지
않는 고정 경로**로 배포된다(`docs/ARCHITECTURE.md`에 적힌 것과 동일한 이유:
`favicon.svg`를 `public/`이 아니라 `src/assets/`에 두는 이유의 반대 케이스 —
매니페스트/아이콘/서비스 워커는 브라우저가 고정 URL로 참조해야 하므로 오히려
해시가 붙으면 안 된다).

- `public/manifest.webmanifest` — 앱 이름·아이콘·`display: standalone`·테마 색.
  **`orientation` 필드는 의도적으로 넣지 않는다** — standalone 설치 앱에서
  `orientation`을 명시하면(`"any"` 포함) 브라우저가 Screen Orientation Lock
  API로 그 값을 강제해, 안드로이드 OS의 화면 방향 고정 설정을 무시하고 항상
  센서 회전을 따라가는 버그가 있었다(실기기 테스트로 확인). 필드를 아예 생략하면
  설치된 앱도 일반 탭처럼 OS의 현재 회전 상태/고정 설정을 그대로 따른다.
- `public/icons/*.png` — 설치 아이콘(192/512/512-maskable/apple-touch-icon).
  `scripts/generate-icons.mjs`가 `src/assets/favicon.svg`와 같은 도형을 좌표
  그대로 코드로 래스터화해서 생성한다(외부 이미지 툴 의존성 없음, Node 내장
  `zlib`로 PNG 인코딩). **favicon 디자인을 바꾸면 이 스크립트의 `BG`/`BAR`/
  `DOT`/`CIRCLES` 상수를 맞춰 고치고 `node scripts/generate-icons.mjs`로 다시
  생성한다.**
- `public/sw.js` — 서비스 워커 본체.
- `src/layouts/Base.astro` — `<link rel="manifest">`, 아이콘/테마 메타 태그,
  그리고 서비스 워커 등록 스크립트.

## 서비스 워커 캐싱 전략

`public/sw.js`의 fetch 핸들러는 요청을 딱 두 갈래로만 나눈다:

1. **내비게이션 요청(HTML 문서)** → *network-first*. 온라인이면 항상 최신
   `index.html`을 받아온다(최신 HTML만이 최신 해시 자산 URL을 담고 있으므로).
   성공한 응답은 매번 캐시에 덮어쓰기 때문에, 별도의 버전 무효화 로직 없이도
   캐시에는 항상 "마지막으로 성공한 최신 응답"만 남는다. 오프라인일 때만 이
   캐시로 폴백한다.
2. **그 외 모든 GET 요청**(`_astro/*` 해시 자산, `/icons/*`, 외부 CDN인
   Bootstrap Icons 폰트 CSS 등) → *cache-first*. 해시 파일명 자산은 내용이
   바뀌면 URL 자체가 바뀌므로 무기한 캐시해도 안전하다 — **파일이 바뀌면 새
   URL이 새로 캐시를 채울 뿐, 옛 URL 캐시 엔트리는 그냥 아무도 참조하지 않게
   되어 자연히 무해해진다.** 캐시를 강제로 비우는 로직이 없어도 최신 파일이
   항상 우선한다는 뜻.

이 두 갈래만으로 "해시 안 붙은 파일은 항상 최신을 확인, 해시 붙은 파일은
영원히 캐시"라는 원칙이 자동으로 지켜진다. 예외 처리를 늘릴 필요가 없다.

### 캐시 이름 · 버전

`SW_VERSION` 상수(현재 `'v1'`)로 캐시 이름을 접두사 처리한다. **콘텐츠가
바뀔 때마다 올릴 필요는 없음** — 위 전략이 이미 콘텐츠 변경을 반영한다. 이
상수는 fetch 핸들러의 전략 자체(예: network-first ↔ cache-first 전환, 캐시
구조 변경)를 바꿀 때만 올린다. 올리면 `activate` 이벤트가 이전 버전 캐시를
전부 지운다.

### 설치/활성화

- `install`: 오프라인 첫 진입 대비로 `/`를 미리 셸 캐시에 데워둔다(실패해도
  설치는 계속 진행 — 실패하면 다음 온라인 방문 때 fetch 핸들러가 채운다).
  `skipWaiting()`으로 바로 활성화 대기 없이 넘어간다.
- `activate`: `SW_VERSION`에 해당하지 않는 캐시를 정리하고 `clients.claim()`.

### 개발 중에는 등록하지 않음

`Base.astro`의 등록 스크립트는 `import.meta.env.PROD`로 감싸져 있다.
`astro dev`에서 서비스 워커가 등록되면 개발 중 파일 변경이 캐시에 가려
헷갈리는 문제가 생기기 때문에, 프로덕션 빌드에서만 등록한다.

## 브라우저 자체 업데이트 절차와의 관계

`sw.js`가 고정 경로(해시 없음)로 나가기 때문에, 브라우저의 표준 서비스 워커
갱신 절차(내비게이션마다 바이트 단위로 `/sw.js`를 다시 받아 비교, 다르면 새
워커를 설치)가 그대로 동작한다. 이 파일 이름/경로를 절대 해시 처리하거나
바꾸면 안 된다 — 그러면 브라우저가 새 워커를 못 찾아 영영 옛 캐시 전략에
갇힐 수 있다.

## 수동 확인 체크리스트

이 저장소의 워크플로우 규칙상 브라우저 테스트(Playwright 등)는 에이전트가
직접 수행하지 않고 사용자에게 요청한다. PWA 관련 변경 후 사람이 직접 확인할
목록:

- 크롬 개발자 도구 → Application → Manifest에서 아이콘/이름 정상 표시.
- 설치 후 아이콘이 홈 화면/앱 목록에 정상 표시(마스커블 아이콘이 잘리지 않는지).
- 설치된 앱을 오프라인 상태(네트워크 끄기)로 재실행해도 정상 로드되는지.
- 파일을 하나 바꾸고 재배포한 뒤, 재접속 시 새 해시 자산으로 갱신되는지
  (오래된 `_astro/*` 캐시 때문에 안 바뀌는 회귀가 없는지).
